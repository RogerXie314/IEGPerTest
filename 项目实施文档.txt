=== 设计方案落实版（由 Copilot 起草，按用户选择 B：先细化模块接口） ===

一、已确认的总体技术选型
- 首选：C# (.NET 7+) + WPF（Windows 桌面原生 GUI），支持高并发、良好与 Windows 集成、方便调用原项目 C++ 代码。
- 备选：Python（PyQt/Tkinter + asyncio）——实现较快但在高并发、Windows 原生集成与打包上劣于 C#。

二、C# vs Python 对比（简要）
- GUI 开发：C# + WPF 原生、开发效率高、与 Windows UX 更一致；Python 需使用 PyQt 等跨平台库，开发和部署复杂度较高。
- 并发/网络：C# 的 TPL 与 socket 支持稳定，高并发性能优；Python 的 asyncio 与多线程可用，但在 CPU/高并发 I/O 场景下需要更谨慎的调优。
- 与原项目互操作：原项目为 C++，C# 可通过 P/Invoke、C++/CLI 或封装网络库快速复用；Python 则需额外绑定层或通过子进程通信。
- 部署与维护：C# 在 Windows 环境的安装包、自动更新与权限控制更方便；Python 需要打包工具（PyInstaller）且依赖管理更麻烦。

结论：综合考虑功能需求（Windows 桌面、高并发网络、与原项目互操作、长期维护），推荐使用 C#。用户已接受 C#，因此后续实现以 C#/.NET + WPF 为主。

三、模块细化与接口定义（草案）
1) 总体模块
   - UI 层（WPF）：配置区、任务区、状态区（心跳/任务表格）、日志分类与白名单上传控件。
   - Core 服务：TaskManager、Worker（RegistrationWorker / HeartbeatWorker / LogWorker / FileUploadWorker）、Network 层。
   - Network 抽象层：ITcpSender, IUdpSender, IHttpSender

2) 数据模型（C# POCO）
   - class TaskModel {
         Guid Id;
         string Type; // "register"|"heartbeat"|"log"|"upload"
         DateTime StartTime;
         int ClientCount;
         int IntervalMs;
         int DurationMinutes;
         TaskStatus Status; // Running|Completed|Failed|Stopped
         int ActiveClientsCount;
         int FailedCount;
      }

   - class ClientModel {
         string ClientId;
         IPAddress Ip;
         ClientState State; // Registered|Unregistered|Error
      }

3) 接口草案
   - interface ITaskWorker {
         Task StartAsync(TaskModel task, CancellationToken ct);
         Task StopAsync();
         event EventHandler<TaskProgressEventArgs> ProgressUpdated;
   }

   - interface INetworkSender {
         Task SendTcpAsync(string host, int port, byte[] payload, CancellationToken ct);
         Task SendUdpAsync(string host, int port, byte[] payload, CancellationToken ct);
         Task<HttpResponseMessage> SendHttpAsync(HttpRequestMessage req, CancellationToken ct);
   }

4) 关键 worker 行为说明
   - RegistrationWorker
      - 并发控制：支持并发数配置（SemaphoreSlim），逐个或并发发送注册请求到目标服务器（TCP/HTTP，按协议）
      - 进度上报：每注册成功/失败更新 `TaskModel.ActiveClientsCount` 和 `FailedCount`，触发 UI 更新事件

   - HeartbeatWorker
      - 定时发送：根据 IntervalMs 周期给每个活跃客户端发送心跳（TCP 到平台 IP），若勾选日志服务器同时发送 UDP 心跳到日志服务器
      - 结束条件：达到持续时长或收到停止请求
      - 性能：使用连接复用与批量发送（当可能时）以降低资源消耗

   - LogWorker
      - 根据日志分类与协议规则选择发送路径（HTTPS 到平台或 UDP 到日志服务器）
      - 支持速率控制（每客户端每秒条数）和批量化发送

   - FileUploadWorker
      - 并发上传白名单文件给指定数量客户端（可用 HTTP multipart 或自定义 TCP 协议），报告成功/失败统计

五、协议/逻辑实现要点
- 未勾选日志服务器：心跳与威胁数据采集走 TCP → 平台 IP；
- 勾选日志服务器：心跳与威胁数据采集仍走 TCP→平台；同时向日志服务器发送 UDP 心跳；其他日志走 UDP→日志服务器。
- 端口测试工具：实现对 TCP/UDP/HTTP 的连通性探测并返回结果（成功/失败/耗时）

- 注册持久化逻辑：
   - 在注册流程中，所有注册成功的客户端记录必须持久化到工程根目录下的 `Clients.log` 文件中（追加写入、结构化每行记录，包含至少：ClientId、IP、注册时间、状态）。
   - 当用户在 UI 中添加“心跳任务”时，系统应首先检查根目录是否存在 `Clients.log`：
      - 若存在：批量读取 `Clients.log` 中的客户端记录，作为心跳任务的目标客户端列表并直接开始心跳上线流程（支持并发配置与速率控制）。
      - 若不存在：提示用户先执行注册任务或手动导入客户端列表。
   - 其它后续任务（白名单上传、日志发送等）均依赖 `Clients.log` 提供的已注册客户端列表；只要根目录存在 `Clients.log`，系统视为已完成注册并可直接使用该文件进行任务调度，直到用户删除该文件为止。
   - 应实现对 `Clients.log` 的读写并发保护（例如文件锁或基于进程互斥的策略），并在 UI 中提供导入/导出与清理（删除）`Clients.log` 的操作入口。

九、项目结构与分类（建议）

为保持实现过程清晰、职责单一并方便维护，建议遵循以下项目结构与分类规范：

- 顶层目录（项目根）
   - `src/`：源代码目录
      - `SimulatorApp/`：WPF 前端（UI）项目（可包含 Views、ViewModels、Resources）
      - `SimulatorLib/`：核心库（Models、Services、Network、Persistence、Utils）
   - `docs/`：设计文档、接口说明、操作说明
   - `tests/`：单元与集成测试代码
   - `scripts/`：构建、发布与辅助脚本（例如 CI helper、迁移脚本）
   - `tools/`：开发或调试用工具/脚本（不随发行包发布）
   - `artifacts/`：构建产物输出目录（默认忽略到 .gitignore）
   - `logs/`：运行时日志（开发环境默认），但关键持久化文件 `Clients.log` 放置在项目根（见下文）
   - `.github/`：CI/CD 工作流与 issue/PR 模板
   - `README.md`, `.gitignore`, `LICENSE` 等根级说明文件

- 代码模块与命名约定
   - `Models`：数据结构和 POCO（例如 `TaskModel`, `ClientModel`）
   - `Services`：任务管理、Worker 调度（`TaskManager`, `RegistrationWorker` 等）
   - `Network`：网络抽象层与具体实现（`ITcpSender`, `IUdpSender`, `IHttpSender`）
   - `Persistence`：持久化相关（读写 `Clients.log`、配置序列化、导入/导出）
   - `UI`（在 `SimulatorApp` 内）：按 MVVM 分层，`Views`、`ViewModels`、`Resources`

- 文件与持久化规范
   - `Clients.log`：放在项目根（或部署目录根），为本项目关键的持久化文件。每行为一条结构化记录（例如 JSON 或 CSV），至少包含 `ClientId, IP, RegisteredAt, Status` 字段。
   - 所有对 `Clients.log` 的访问必须有并发保护（文件锁或进程互斥）并使用原子追加/替换策略以防止数据损坏。
   - `logs/` 目录仅用于运行时日志（按日期分割），关键持久化数据集中放在根目录或 `data/` 可配置位置。

- 分支/版本控制策略（建议）
   - 主分支 `main`：稳定可发布代码
   - 开发分支 `develop`：日常合并的集成分支
   - 功能分支 `feature/xxx`：每个任务或模块一个分支，完成后合并到 `develop`
   - 发布分支 `release/*` 与热修复 `hotfix/*` 按需使用

- 其他建议
   - 在 `SimulatorLib` 中严格定义接口并编写单元测试，UI 层应尽量只做展示与事件触发。
   - 将与原项目互操作（C++）的部分封装为单独模块（`Interop`），并在项目初期明确调用方式（P/Invoke 或 C++/CLI）。

注：本规范作为实现参考，具体目录与命名可根据实际团队偏好微调，但请保持模块边界清晰、持久化与日志位置固定（`Clients.log` 在根目录）。
