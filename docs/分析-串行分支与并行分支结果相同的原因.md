# 分析：串行分支（test/old-tool-log-dispatch）与并行分支结果相同的原因

> **背景**：`test/old-tool-log-dispatch` 分支将 `LogWorker` 的 TCP+EPS 发送模式改为
> 单一循环顺序遍历所有客户端（模仿 C++ 老工具串行模型），但实测结果与并行版本并无本质区别。
>
> 本文从代码层面分析可能的原因，**只分析，不改代码**。

---

## 一、串行分支做了什么

提交 `eef7a40`（2026-03-05）在 `LogWorker.cs` 中加入了一个**单一调度循环**：

```csharp
// 原并行：每客户端独立 Task.Run(async () => { while(...) { await Send...; } })
// 串行替换：
while (!ct.IsCancellationRequested)
{
    var roundStart = Stopwatch.GetTimestamp();

    for (int ci = 0; ci < clients.Count; ci++)   // 顺序遍历所有客户端
    {
        await SendLogTcpLikeExternalAsync(clientId, ..., pt, ct);
    }

    // 补满 intervalMs（若一轮发完还有剩余时间才睡眠）
    var waitMs = (int)(intervalMs - elapsedMs);
    if (waitMs > 10)
        await Task.Delay(waitMs, ct);
}
```

设计意图：对齐 C++ 老工具的 "一个日志线程，依次 SendData，完成后 Sleep(intervalMs)" 模型，
消除多 Task 并行突发带来的写锁争用，验证 "C# async 并发" 对心跳稳定性的贡献。

---

## 二、实测结果（来自 BUG-2026-001 文档 Step 7）

| 测试 | EPS | 时长 | 日志失败 | TCP 断连 | HB 回包骤降 |
|---|---|---|---|---|---|
| 串行分支（Step 7） | 600 EPS | 12.4 h | **0 次** | **0 次** | **170 次**（均自恢复） |
| 并行分支（Step 2） | 2000 EPS（2台合计） | ~5 min | 7965 次 | 大量 | 断连风暴 |
| 并行分支（Step 5） | 500 EPS | 15 min | 0 次 | 0 次 | 无 |
| 并行分支（Step 6） | 750 EPS | 短测 | 有 | 有 | 有 |

结论：**串行版本在 600 EPS 时已比并行版本在 1000 EPS 时稳定得多。**
但串行版本在同等 EPS（约 750+）下依然会出现掉线；
在 600 EPS 下仍有"每 4.4 分钟一次的回包骤降"——"结果和这个没什么区别"的感觉
来自这 170 次骤降依然存在，而且骤降特征（平台静默 30~90 秒再自恢复）与之前并行版本的
现象在形态上相同。

## 三、为什么串行版本结果与并行版本相近

### 原因 1：根本原因在平台侧，客户端发送模型无法改变（**最主要原因**）

BUG-2026-001 Step 3 已通过平台日志确认（`假说 D` ✅）：

```
日志发送开始 16:35:07
首批 PolicyDrain 16:36:37 = 16:35:07 + 精确 90 秒
→ 平台 HB watchdog 超时阈值 = 90 秒
→ 根因：平台的日志 I/O 处理占满了 HB handler 资源，HB 响应被饥饿 90s 触发 PolicyDrain
```

这意味着：**只要向平台发送的日志总量超过其处理能力阈值，平台就会停止响应 HB，
90 秒后触发 PolicyDrain**。这与客户端是并行发送还是串行发送完全无关——
平台看到的总日志流量是一样的。

串行调度器同样向 600 个 TCP 连接各发 1 条/秒 = 600 EPS，
平台侧的 HB 处理瓶颈完全相同。

---

### 原因 2：C# 串行循环依然是"一轮 600 次无间隔"的爆发

C++ 老工具日志线程的实际节奏（来自文档）：
```
日志线程: 复用同一 g_sock[i]，3 次 SendData + Sleep(50)，不 recv
```

即：**发 3 条，强制睡 50ms，再发 3 条**。这在发送动作之间插入了显式的 50ms 间隙，
让平台的 HB 处理线程在间隙内腾出资源。

C# 串行调度器的实际节奏：
```
for ci in 0..N: await Send(ci)   // 连续 N 次 await，之间无显式间隔
// 一轮结束后，若 elapsedMs < intervalMs 才睡眠
```

当客户端数量较多（如 600）且每次 `SendLogTcpLikeExternalAsync` 耗时约 1~2ms 时，
一轮 600 次发送合计约 600~1200ms。若 `intervalMs = 1000ms`（1 EPS/客户端），
则 `waitMs = 1000 - 1000 ≤ 0`，**没有睡眠时间**，下一轮立即开始。

结果：从平台视角看，C# 串行和 C# 并行都是**近乎连续**的 600 EPS 数据流，
没有真正的"发 N 条、休息一段、再发"节奏。

---

### 原因 3：C# 的 `await` 与 C++ 的 `Sleep(50)` 语义不同

| 特性 | C++ `Sleep(50)` | C# `await Task.Delay` |
|---|---|---|
| 执行模型 | OS 内核级阻塞，精确释放 CPU 50ms | 将协程挂起，线程返回线程池但可立即执行其他 await |
| 调度保证 | OS scheduler 强保证 ≥50ms 不执行 | .NET 线程池调度，实际延迟可能为 15~20ms（Windows 时钟精度） |
| "留空窗口"效果 | 50ms 内此线程完全不发送，平台有充足时间响应 HB | `waitMs` 可能为 0（不睡眠），或精度不足，窗口效果差 |

C++ 的 `Sleep(50)` 在操作系统层面强制制造了一个 50ms 的无发送窗口，
这个窗口足以让平台的 HB handler 完成响应。C# 的串行调度没有等效的显式休眠。

---

### 原因 4：`intervalMs` 语义差异 —— C# 是"每客户端 1条/周期"，C++ 是"整组客户端 3条/Sleep"

C# 串行调度中：
```csharp
intervalMs = 1000 / messagesPerSecondPerClient;  // 1 EPS → intervalMs=1000ms
// 含义：每 intervalMs 发送1轮（覆盖所有客户端各一条）
```

当 `clients.Count = 600`，`intervalMs = 1000ms` 时，
实际平均发送速率 = 600 / 1秒 = 600 EPS（与期望一致）。
但这 600 条在单轮内集中发出（约 600~1200ms 内），**无法均匀分散**。

C++ 老工具的发送逻辑并非"每秒每客户端1条"，
而是由 `Sleep(50)` 自然产生约 20批次/秒、每批 3 条的发送节奏，
配合多线程（HBClientCount_InEveryThread=10）的线程级并发，
实际上对平台的瞬时写压力要低于 C# 的单线程集中式发送。

---

### 原因 5：`SemaphoreSlim` 写锁超时仍可能偶发性饥饿 HeartbeatWorker

即使在串行模式下，`SendLogTcpLikeExternalAsync` 仍然调用：
```csharp
bool lockAcq = await _streamRegistry.AcquireWriteLockAsync(clientId, 200, ct);
```

串行调度在对客户端 `ci` 发送时会持有其写锁。
如果某个 `WriteAsync` 因 TCP 缓冲区背压而阻塞（时间 > 200ms），
则 `HeartbeatWorker` 尝试获取同一 clientId 的写锁时依然会超时（`LockBusy`），
跳过本次心跳。

不过串行模式下每个 clientId 在一轮内只被访问一次，锁持有时间更短，
此情况比并行模式少见。但在 EPS > 阈值（700~750）时，
TCP 缓冲区背压概率增大，依然可能出现偶发性 LockBusy。

## 四、总结对比

| 维度 | C# 并行 | C# 串行（test/old-tool-log-dispatch） | C++ 老工具 |
|---|---|---|---|
| 客户端发送模型 | N 个并发 Task | 单循环顺序遍历 | 多 OS 线程，每线程 10 客户端 |
| 发送节奏 | 各自独立、近乎连续 | 一轮 N 条、无显式间隙 | 3条 + Sleep(50ms) 循环 |
| 对平台的日志总流量 | EPS = N × eps_per_client | **相同** | 实际可能更低（Sleep 限速） |
| 平台侧 HB 处理瓶颈 | 受日志流量饥饿 | **相同受限** | 有 50ms 间隙，HB 处理有窗口 |
| 写锁饥饿（LockBusy） | 高 EPS 时频繁出现 | 较少，但仍可能偶发 | 不存在（无 semaphore） |
| 根本结论 | 客户端侧无问题，平台侧瓶颈 | **相同结论** | 发送节奏让平台有喘息空间 |

**一句话原因：串行版本解决了客户端侧的写锁争用问题，但没有改变发往平台的总日志流量，
平台侧的 HB handler 受日志 I/O 饥饿依然存在，90s watchdog 在 EPS 超过阈值时依然会触发，
这就是为什么两者结果没有本质区别。**

---

## 五、要真正对齐 C++ 稳定性的方向（仅供参考，不改代码）

若要在 C# 侧复现 C++ 工具的稳定性，理论上需要：

1. **在每批发送后插入显式的 Sleep/Delay**（对齐 C++ `Sleep(50)`），
   主动制造"无发送窗口"让平台 HB handler 有喘息时间。
2. **降低总发送速率至平台承受阈值以下**（实测约 600 EPS 边界，500 EPS 完全稳定）。
3. 或等待**平台方修复 HB 处理优先级**（提高 HB 响应独立于日志 I/O 的调度优先级，
   或放大 watchdog 阈值 > 90s，或引入日志接收端背压机制）。
