## 现象

- 启动 2000 个客户端，心跳 + 日志，大量客户端离线，同时 UI 显示日志发送失败
- 老工具（WLServerTest C++）同等配置可稳定运行

> ⚠️ 注：「纯 HB 离线」为文档错误描述，已勘误。实测纯 HB 运行 30 分钟完全稳定。

---

## 背景 & 架构对比

### 老工具（WLServerTest，C++）

```
HBClientCount_InEveryThread = 10
→ 2000 客户端 = ~200 个 OS 阻塞线程
→ 每线程: connect → 发 HB → recv HB/LogACK → Sleep(30000)
→ 日志线程: 复用同一 g_sock[i]，3 次 SendData + Sleep(50)，不 recv
→ 日志 ACK 由 HB 线程的 RecvHeartBeatBack_TCP 的 else 分支消费
→ Sleep(30000) 精准，OS 级强保证
```

### 当前 C# 实现

```
每客户端 1 个 HeartbeatWorker Task + 1 个 DrainTask + 1 个 LogWorker Task
→ 2000 客户端 = 6000+ 并发 Task，共用 .NET 线程池（默认 ~core 数）
→ HeartbeatWorker: Task.Delay(30s)，线程池有压力时有抖动
→ LogWorker: AcquireWriteLockAsync(200ms) + WriteAsync → FlushAsync，不读 ACK
→ DrainTask: Task.Run() 无限 ReadAsync，消费所有入方向字节
```

## 根因假说（待验证）

### ~~假说 A：Task 调度抖动导致 HB 超时~~ ❌ 已排除
- 2026-03-05 实测：2000 客户端纯 HB 运行 **30 分钟**，全程稳定无掉线
- Task.Delay(30s) 调度精度在此规模下不是问题

### ~~假说 B：日志写操作干扰 HB stream~~ ❌ 已排除
- 2026-03-05 Step 2 实测：test3- 机器**仅运行心跳**（无日志任务），却在 16:40:46 同时出现「平台回包:0 / 服务端关闭」
- 机器1(test2-)、机器2(xrg-)、机器3(test3-) 三台机器在 **16:40 前后同时**被平台断连
- 断连原因均为服务端主动行为：「平台断session」「服务端关闭」
- 结论：问题**根源在平台侧**，与客户端写锁无关

### ~~假说 C：平台连接数/处理能力上限~~ ❌ 已排除（描述不准确）
- 断连类型为 `PolicyDrain`（平台有序主动关闭），而非连接数满/OOM/进程重启
- 日志中无 `too many open files` / OOM / 守护进程崩溃迹象
- 但「平台侧问题」方向正确，细化见假说 D

### 假说 D：平台 HB 处理被日志 I/O 竞争阻塞，90s watchdog 触发 PolicyDrain ✅ **已确认（Step 3）**

#### 证据（来自 2026-03-05 平台日志）

| 时间 | 事件 | 来源 |
|---|---|---|
| 15:52:06~07 | 1000 test2 客户端 TCP session 建立 | heartbeat_monitor |
| 16:35:07 | 日志发送开始；平台最后一次回 HB 包 | heartbeat_monitor + logsend-packets |
| 16:36:37.416 | **首批 PolicyDrain**：test2-18/26/20/8 等，上次回包均为 16:35:07（**精确 90 秒**） | heartbeat_monitor |
| 16:37:07 | 回包骤降 511（997→486），TCP未回包 514；断连风暴高峰 | heartbeat_monitor |
| **16:37:08 UTC** | logsend failures 开始，`reason=non_ok_response`，持续 7965 条 | logsend-failures |
| 16:37:16 | Kafka consumer group `usm` rebalance（PolicyDrain 引发连锁） | journalctl |
| 16:37:37 | 回包 229，TCP未回包 771 | heartbeat_monitor |
| 16:40:01 | CRON: `sysctl vm.drop_caches=1` + `timer.sh`（影响 test3 纯HB机） | journalctl |
| ~17:10 | logsend failures 持续至测试结束（循环断连/重连/再断） | logsend-failures |

#### 断连模式分析

```
上次回包 = 16:35:07（日志发送开始时刻）
服务端关闭类型 = PolicyDrain（平台主动有序关闭，非超时/RST）
触发时间 = 上次回包 + 90 秒 ← 平台 HB watchdog 超时阈值
```

所有被 PolicyDrain 的 session：
- **上次 HB 回包均停留在 16:35:07**（日志发送开始的瞬间）
- 此后平台完全停止对这些 session 的 HB 做出应答
- 90 秒后 watchdog 触发 PolicyDrain

#### 根本机制

1. 1000 个客户端同时发送 TCP 日志包（2000 EPS 合计）
2. 平台日志接收端 I/O / 处理队列被日志流量占满
3. 同一 session 的 **HB 响应被日志处理阻塞/饥饿**，停止应答
4. 平台内部 HB watchdog（**90 秒超时**）检测到 1000 个 session 无心跳
5. 触发批量 `PolicyDrain`（雪崩式，在 ~30 秒内完成 1000 个 session 的关闭）
6. PolicyDrain → Kafka consumer group `usm` 感知连接断开 → rebalance（PolicyDrain 后 9 秒）
7. 客户端重连后再次发日志 → 再次 PolicyDrain → 循环，导致持续 35 分钟的 logsend 失败

#### 为何 test3（纯HB对照机）也断连
- test3 在 **16:40:01** CRON 触发 `sysctl vm.drop_caches=1` + `timer.sh` 时断连
- 此时平台可能执行了 session 全量清理/维护操作
- test3 断连原因与 test2/xrg 不同（是 CRON 触发，不是 HB watchdog）

### `6ddf31d` 为何是错的（已 revert）
之前误判老工具用独立短连接发日志，实际老工具：
- `g_sock[i]` = HB 建立的持久 socket，日志线程复用它
- 日志线程**不 recv**，ACK 由 HB 线程的 `else` 分支消费
- C# 当前实现已对齐此行为（`LogWorker` 不 recv，DrainTask 消费入方向）

### 路线 D（当前路线）：等待平台研发修复 HB 优先级问题
- **客户端无需改动**：已正确按老工具行为实现(同 session 发日志+HB，DrainTask 消费回包)
- **平台需修复**：
  - 方案①：HB 回包不依赖日志处理完成，独立线程/优先级处理
  - 方案②：调大 HB watchdog 超时阈值（90s → 更长），容忍日志 I/O 抖动
  - 方案③：日志接收端引入背压，写满时暂停接收而不断开 HB session
