## 现象

- 启动 2000 个客户端，心跳 + 日志，大量客户端离线，同时 UI 显示日志发送失败
- 老工具（WLServerTest C++）同等配置可稳定运行

> ⚠️ 注：「纯 HB 离线」为文档错误描述，已勘误。实测纯 HB 运行 30 分钟完全稳定。

---

## 背景 & 架构对比

### 老工具（WLServerTest，C++）

```
HBClientCount_InEveryThread = 10
→ HB 线程: 每线程管 10 个客户端，2000 客户端 = 200 个 HB OS 线程
  每线程: connect → 发 HB → recv HB/LogACK → Sleep(30000)
→ 日志线程: 每客户端 1 个独立 OS 线程（THREAD_PRIORITY_TIME_CRITICAL）
  每线程: do { SendData(g_sock[i], payload×3种); Sleep(interval); } while(1)
  复用 HB 建立的 g_sock[i]（无锁），不 recv，ACK 由 HB 线程消费
→ 2000 客户端 = 200 HB线程 + 2000 日志线程，共 2200 个 OS 线程
```

### 当前 C# 实现

```
每客户端 1 个 HeartbeatWorker Task + 1 个 DrainTask + 1 个 LogWorker Task
→ 2000 客户端 = 6000+ 并发 Task，共用 .NET 线程池（默认 ~core 数）
→ HeartbeatWorker: Task.Delay(30s)，线程池有压力时有抖动
→ LogWorker: AcquireWriteLockAsync(200ms) + WriteAsync → FlushAsync，不读 ACK
→ DrainTask: Task.Run() 无限 ReadAsync，消费所有入方向字节
```

## 根因假说（待验证）

### ~~假说 A：Task 调度抖动导致 HB 超时~~ ❌ 已排除
- 2026-03-05 实测：2000 客户端纯 HB 运行 **30 分钟**，全程稳定无掉线
- Task.Delay(30s) 调度精度在此规模下不是问题

### ~~假说 B：日志写操作干扰 HB stream~~ ❌ 已排除
- 2026-03-05 Step 2 实测：test3- 机器**仅运行心跳**（无日志任务），却在 16:40:46 同时出现「平台回包:0 / 服务端关闭」
- 三台机器在 **16:40 前后同时**出现部分掉线，且 C# 发日志这个动作是触发点
- 结论：写锁设计方向正确，干扰 HB stream 的假说排除

### ~~假说 C：平台处理能力上限~~ ❌ 已排除
- C++ 老工具同一平台可稳定打 **4000 EPS**，短时达到 **6000 EPS**，3000 HB 心跳全程稳定
- 平台不是瓶颈，问题在 **C# 发送侧**

> **C++ 4000 EPS 测试配置说明**：6 台主机 × 500 客户端 = 3000 个心跳连接，全程稳定；
> 其中 4 台主机开启日志发送，合计打出 **4000 EPS**（非 2×1000 EPS）。

### 假说 D：C# 发送能力差距 ⭐ 当前主要方向
- 同一平台：C++ 6 台×500 客户端 4000 EPS 稳定，C# 1000 EPS 就开始出现掉线，差距约 4 倍
- C# 和 C++ 的 JSON 字段结构、协议封装（Zlib 压缩，无加密）已确认一致，不是格式问题
- **待查方向**：C# per-client Task.Run 路径在高 EPS 下的实际发送成功率与发送能力上限

### `6ddf31d` 为何是错的（已 revert）
之前误判老工具用独立短连接发日志，实际老工具：
- `g_sock[i]` = HB 建立的持久 socket，日志线程复用它
- 日志线程**不 recv**，ACK 由 HB 线程的 `else` 分支消费
- C# 当前实现已对齐此行为（`LogWorker` 不 recv，DrainTask 消费入方向）

## 修复路线（待定）

### ~~路线 A：HeartbeatWorker Task 批量化~~ — 已排除（Step 1 验证 HB 本身稳定）

### ~~路线 B：拉长写锁超时~~ — 已排除（写锁是 per-clientId 独立的，2000 把锁互不竞争）

### ~~路线 C：增大线程池下限~~ — 已排除（问题是发送能力，不是调度延迟）

### 路线 D：分析 C# per-client Task.Run 路径的实际 EPS 上限
- 在 main 分支（干净的 per-client Task 实现）测试不同 EPS 下的实际发送成功率
- 对比 C++ 和 C# 在相同客户端数下的实际到达平台包数

### ✅ 路线 E（最终修复）：LogWorker 改为线程启动一次，每轮发 3-5 条后退出
- 不再 per-client 持续跑 Task，改为线程级批量发送（每次启动发一轮 3-5 条）
- 消除了高并发下 Task 长期持有连接写锁产生的资源竞争
- **验证结果**：1000 EPS 持续压测 24 小时，心跳全程无掉线、无掉线后不上线现象
