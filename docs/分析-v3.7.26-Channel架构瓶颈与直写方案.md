# v3.7.26 Channel 架构瓶颈分析与直写方案

> 分析日期：2026-03-18
> 分析模型：Claude Opus 4.6
> 分析版本：v3.7.26（fix/threat-hit-ratio 分支）

---

## 现象

v3.7.26 跑 3600 EPS，平台 `wl_limit` 统计的实际入库量远低于工具侧报告值。
截图中平台每分钟计入量约 1000 条/s 级别，工具侧声称 3600 成功——差距约 3.6 倍。

---

## 结论：C# 不是问题，Channel 架构才是

C# 完全可以达到 C++ 的吞吐量。问题在于 v3.7.16 引入的 Channel 架构从根本上改变了数据包的发送模式。

---

## 根因分析

### 1. "成功"计数的含义错位

```csharp
// LogWorker.cs — allThreatTcp 路径
var typeFailR = TryEnqueueLog(c.ClientId, pt);
if (typeFailR == null) IncSuccess();  // ← 入队成功就算"成功"
```

`IncSuccess()` 统计的是"放进 Channel 的数量"，**不是** TCP 实际写到 socket 的数量。
真正的发送发生在 HeartbeatWorker 的 drain 循环里，那里没有成功/失败计数器。

### 2. 数据路径对比（关键差异）

| 方面 | C++ 老工具 | C# v3.7.26 |
|---|---|---|
| **数据路径** | Log 线程 → **直接** `send(sock)` → 网卡 | LogWorker → Channel → HeartbeatWorker drain → `WriteAsync` → 网卡 |
| **发送节奏** | 每客户端线程 `Sleep(1000ms)` → 3包/100ms → 900ms idle | 入队后等 drain 333ms 间隔逐包写出 |
| **时钟精度** | Win32 `Sleep()` + 阻塞 `send()` = 硬同步 | `Task.Delay(333)` = ~15ms timer 抖动 + 线程池调度延迟 |
| **HB/Log 关系** | 两个线程各自写同一 socket，OS Winsock 内核串行 | HeartbeatWorker 唯一写者，HB 发送**阻塞** drain |

### 3. 平台限速触发机制

drain 循环在 HB 写完后的 `intervalMs - 100ms` 窗口内以 333ms 间隔逐包发送。
但 `Task.Delay(333)` 的实际分辨率约 15ms，1200 个客户端的 drain 唤醒在某些 timer tick 上扎堆，
形成微观突发（micro-burst），触发平台 `wl_limit` 限速器。

### 4. 16 个版本迭代总结

| 阶段 | 版本 | 做了什么 | 为什么没解决 |
|---|---|---|---|
| 连接修复 | v3.7.3~v3.7.10 | 去 CT、KeepAlive、NOREGISTER | 治标：修了连接误断 |
| 锁方案 | v3.7.11~v3.7.15 | 锁超时/无超时/强制重连 5 种 | 在"多写者+锁"内打转 |
| **Channel** | **v3.7.16** | **HeartbeatWorker 唯一写方** | **消除锁竞争，但引入入队≠发送** |
| 调参 | v3.7.19~v3.7.22 | drain 限速 → 移除 → 再限速 | 参数调不到本质 |
| 修 bug | v3.7.24~v3.7.26 | 断线丢弃 + SessionStale + drain 对齐 | 修了3个bug，Channel延迟仍在 |

根本矛盾：Channel 为解决 "两个 Writer 争一根 TCP 连接" 引入。
但 C++ 从不需要解决这个问题——靠 OS `send()` 原子性搞定。

---

## 解决方案：直写架构（Direct Send）

### 核心思想

放弃 Channel 间接层，LogWorker **直接写 TCP stream**，还原 C++ 数据路径。
用 `SemaphoreSlim(1,1)` 最小化锁持有时间。

与 v3.7.13 之前的锁方案本质不同：
- v3.7.13 时代是**无限速 burst**（数千 pps/client），锁持有时间长达秒级
- 现在是**3 pps/client**（已被绝对 deadline 严格限速），每次小包 WriteAsync 微秒级

### 架构变更

```
之前：LogWorker → TryEnqueueLog → Channel<byte[]> → HeartbeatWorker drain → WriteAsync → TCP
之后：LogWorker → await writeLock → WriteAsync → TCP    (直接)
      HeartbeatWorker → await writeLock → WriteAsync → TCP  (直接)
```

### 为什么不会重蹈 v3.7.11~v3.7.15 的覆辙

| 条件 | v3.7.11~v3.7.15 | 新方案 |
|---|---|---|
| 每客户端发包速率 | 无限速 burst（数百 pps） | 严格限 3 pps（绝对 deadline 已验证） |
| 锁持有时间 | 秒级（TCP 背压下 WriteAsync 阻塞） | 微秒级（3pps 远低于 TCP 带宽） |
| 背压场景 | 高 EPS burst 填满 TCP send buffer | 不可能（3 小包/s/连接 vs 数 MB/s 带宽） |
| 锁竞争概率 | 接近 100%（burst 时） | < 0.5%（1 HB/30s vs 3 Log/s） |

### 改动文件

| 文件 | 变更 |
|---|---|
| `HeartbeatStreamRegistry.cs` | 新增 `AcquireWriteLockAsync` / `ReleaseWriteLock`；保留 Channel 作为 fallback |
| `LogWorker.cs` | allThreatTcp 路径：用 `DirectSendAsync` 替代 `TryEnqueueLog`，直接写 TCP |
| `HeartbeatWorker.cs` | HB 写入走 writeLock；drain 循环仅在有 Channel 积压时才运行（兼容旧路径） |
