
```powershell
mv Clients.log Clients.log.bak
```

**失败日志位置**（程序运行目录下）：

|---|---|
| `logs/logsend-failures-*.log` | 日志发送失败分类统计（≤ 50 MB） |

| `logs/throughput-*.log` | 可选：实测 EPS 统计（成功/失败每秒速率），需调用 `logWorker.Metrics.Enable()` 启用 |
**SSH 平台日志收集（`TcpDiagWindow` → "SSH 日志收集" 区）**：

1. 填入平台 SSH 地址/端口/账号/密码，点 **"收集日志"**
2. 程序自动：RTT 偏差检测 → `su root` → `find /root/logs -newer .ts_marker`（递归）→ 小文件直接 SFTP、大文件服务端 `awk` 时间戳过滤 → 收集 `/var/log/messages`
3. 单个文件失败自动跳过（`⚠ xxx 跳过: ...`），不影响其他文件
4. 输出目录：`debug_collect_yyyyMMdd_HHmmss/`（程序工作目录下）

| 输出文件 | 来源 |
|---|---|
| `<subdir>/*.log` | `/root/logs/**`（保留子目录结构）|
| `var_log_messages` | `/var/log/messages`（内核级 socket/fd 错误）|
| `heartbeat_monitor_*.log` | 本地心跳断线日志 |
| `logs/*.log` | 本地 logsend / logsend-failures / logsend-https 等 |
| `Clients.log` | 注册结果快照（不按时间过滤）|

### v3.7.54 — 2026-03-20：修复 NativeEngine 首次建连被误计为重连

**现象**：心跳刚启动时日志显示 `在线:60 重连:60`，在线多少重连就多少，正常运行中 `重连` 字段无意义。

**根因**：`HBThreadProc` 第一段"初始建连"里 `slot.sock = CreateConnection(...)` 后直接 `s_reconnects++`，把首次上线也算进了重连计数器。

**修复**：去掉初始建连处的 `s_reconnects++`；`reconnects` 字段现在只在以下真正的重连场景累加：
- `HBDoSendRecv`：send 失败 → 关旧 socket → 重建连接
- `HBDoSendRecv`：收到 NOREGISTER(cmdId=18) → 重注册后重建连接
- 主循环：检测到 `slot.sock == INVALID_SOCKET` → 断线重建连接

### v3.7.51 — 2026-03-20：修复 Log 线程两处与老工具不一致导致的掉线和 EPS 偏低

**现象**：400 客户端日志运行约 5 分钟后大量掉线，EPS 持续偏低。

**Bug 1：LogThreadProc 每轮读取 `slot.sock`（v3.7.50 意外退回了此问题）**
- 根因：`SOCKET sock = slot.sock` 写在 `while` 循环内部，每轮重读当前值
- 老工具行为：`pHeapArgsForAllThread_MsgLog->sock = g_sock[i]` 在线程创建前设定，`pHeapArgs->sock` 之后永不更新（值拷贝语义）；HB 重连后 log 线程透明，最多失败继续
- 当前错误行为：HB 重连 → `slot.sock = newSock`（未经首次 HB 认证）→ log 线程下轮立即发 → 服务端关闭未认证连接 → 级联断线
- 修复：启动时等待第一个有效 sock 后取一次快照，之后循环内不再读 `slot.sock`（v3.7.48 正确修复，本次重新落实）

**Bug 2：缺少 `THREAD_PRIORITY_TIME_CRITICAL`（v3.7.50 错误移除）**
- v3.7.50 记录"老工具无此设置"是错误判断；老工具 `WLServerTestDlg.cpp:2170`：`AfxBeginThread(..., THREAD_PRIORITY_TIME_CRITICAL, ...)`
- 移除后 400 线程被 OS 调度器抢占，`Sleep(1000)` 实际拉长 20-50%，EPS 系统性偏低
- 修复：`CreateThread` 后立即 `SetThreadPriority(h, THREAD_PRIORITY_TIME_CRITICAL)`

### v3.7.48 — 2026-03-19：Log 线程 socket 快照——对齐老工具值拷贝语义

**现象**：平台 UI 显示在线数持续减少。  
**根因**：`LogThreadProc` 用 `ClientSlot& slot`（引用）访问 `slot.sock`，每次迭代都读当前值。当 HB 线程重连后更新 `slot.sock = newSocket`，log 线程立即向尚未完成注册的新 socket 发送日志，服务端因收到未注册 socket 的报文而关闭连接，形成重连级联。  
**修复**：线程启动时取一次快照 `SOCKET sock = slot.sock;`，之后始终用这个常量句柄，完全对齐老工具 `pHeapArgs->sock` 的值拷贝语义。

### v3.7.43 — 2026-03-19：完全对齐老工具线程模型

**现象**：v3.7.41 跑出的 EPS 波形仍然缓慢爬坡+振荡，老工具波形平坦稳定；两者都用 CreateThread 创建 OS 线程，但行为差异明显。

**根因分析（两个独立问题）**：

1. **启动时间 250s vs 25s（导致缓慢爬坡）**
   - 老工具：`AfxBeginThread` 每次启动一个线程，每线程管理 10 个客户端，50 线程 × 500ms = **25 秒**全部上线
   - 我们：500 线程 × 500ms = **250 秒**，log 线程跟随上线，EPS 缓慢爬坡 250 秒才到峰值

2. **400 个 log 线程同时启动——雷群效应（导致振荡）**
   - `NE_StartLogSend` 一次创建全部 400 个 log 线程，第一轮无 Sleep，全部同时爆发
   - 之后每 1 秒同步爆发一次 → 平台 TCP 缓冲区周期性拥塞 → 部分 send 返回 WSAEWOULDBLOCK → 周期性 EPS 下降 → 波形振荡

**修复**：
- **HB 线程改为 10 客户端/线程**（`HB_CLIENTS_PER_THREAD = 10`）：50 线程 × 500ms = 25 秒上线，对齐老工具
  - 抽取 `HBDoSendRecv()` 辅助函数，`HBThreadProc` 接收 `HBGroupArg{startIdx, count}`，顺序处理本组所有客户端
  - `NE_StartHeartbeat` 按步长 10 创建线程，句柄存在每组第一个 slot
- **log 线程加初始相位错开**：`staggerMs = intervalMs × idx / logClientCount`
  - Thread 0 不等待，Thread 199 等 500ms，Thread 399 等 999ms
  - 400 个线程均匀分布在 1s 窗口内，彻底消除雷群效应

### v3.7.43 — 2026-03-19：全面对齐老工具线程模型（一次性补全所有差异）

**背景**：v3.7.40–v3.7.42 均为症状驱动的零散修补，未做系统性差异审计。本版完整阅读老工具 `ThreadFunc_HeartbeatSend_New` 和 `ThreadFunc_MsgLogSend` 源码，识别并一次性修复全部剩余差异。

**与老工具的差异清单及修复**：

| 差异 | 老工具行为 | 修复前 | 修复后 |
|------|-----------|--------|--------|
| HB 线程分组 | 10 客户端/线程，500÷10=50线程，25秒启动 | 1客户端/线程，500秒启动 | 10客户端/线程 ✅ |
| Log 线程 Sleep 位置 | 先发完，**再** Sleep | **先** Sleep（导致第一轮延迟，后续时序偏移）| Sleep 移到循环末尾 ✅ |
| Log 线程优先级 | `THREAD_PRIORITY_TIME_CRITICAL` | `THREAD_PRIORITY_NORMAL` | 改为 `TIME_CRITICAL` ✅ |
| Log 线程启动时 socket 可用性 | 老工具 HB 完全跑起来后用户再手动启日志，socket 全部就绪 | 与 HB 同时启动，早期 socket 未就绪导致空转失败 | 启动后轮询等待 `slot.connected==1` 再进入发送循环 ✅ |
| 雷群效应（Thundering Herd） | 老工具因 HB 建连时序自然错开 | HB 全部同时启动，Log 同时爆发 | 在 socket 就绪后显式 stagger ✅ |

**预期效果**：
- Log 线程第一轮立即发送（无多余初始延迟）
- 高优先级线程减少 OS 调度抖动
- 启动阶段不再有 400 个线程同时失败重试
- 整体 EPS 更稳定，与老工具基准（~1000 EPS）一致

### v3.7.42 — 2026-03-19：修复统计轮询在心跳启动后250秒才开始的问题

**根因**：`NE_StartHeartbeat()` 内部 for 循环每创建一个线程 `Sleep(connectGateMs=500ms)`，500个客户端共阻塞 **约250秒**。统计轮询 `Task.Run` 写在 `StartHeartbeat()` 之后，导致整个门控等待期间 UI 上"在线客户端"等字段全为 0。

### v3.7.29 — 2026-03-18：修复 LogWorker 空转紧循环导致 CPU 饥饿 + ThreadPool 雪崩

**现象**：v3.7.28 测试 wl_limit 显示平台只收到 ~944 EPS（目标 1200），其他主机心跳仍然断线，与 v3.7.27 无改善。

**根因链（ThreadPool 雪崩 → 连接风暴 → 平台级联射）**

1. LogWorker spread (10s) < HeartbeatWorker spread (30s)，末尾 ~200 客户端的 LogWorker Task 在 HB 连接建立前就开始发送
2. `DirectSendAsync` 返回 `"disconnected"`（stream 不存在）→ 内层 `break` → 外层 `for` 立即下一 msgIdx
3. 绝对 deadline 已过（失败瞬间完成）→ **无任何等待** → 紧循环每轮都做 JSON 构建 + Zlib 压缩
4. 200 个 Task 全速 CPU 压缩 → ThreadPool 饥饿（默认 MinThreads=CPU核数）
5. HeartbeatWorker Task 排不上线程 → HB 超时 → 平台关连接
6. 更多断线 → 更多空转 → 雪崩加剧 → 平台接受队列满载 → 波及其他主机心跳

**修复 1：断开时等待 interval（LogWorker.cs）**

```csharp
if (typeFailR == "disconnected" || typeFailR == "no_lock")
    await Task.Delay(Math.Max(intervalMs, 1000), ct);
```

当 stream 不存在时，LogWorker 等待一个 interval（对齐 C++ MsgLog 线程 Sleep(interval)），而不是空转烧 CPU。同时修复 allThreatTcp 和单分类威胁 TCP 两条路径。

**修复 2：ThreadPool.SetMinThreads(500, 500)（App.xaml.cs）**

``` csharp
// 400 客户端 × (HB + drain + Log) = 1200 并发 Task
```

.NET ThreadPool 默认 MinThreads = CPU 核数（~8），新线程创建速率仅 1/500ms。
1200 Task 同时启动时，只有 8 个能跑，其余排队等待线程注入，导致大量异步回调延迟。
提前设置 500 min threads 确保 Task.Delay/WriteAsync 回调立即有线程可用。

### v3.7.28 — 2026-03-18：修复 v3.7.27 直写架构两个严重 bug

**现象**：v3.7.27 测试 400×1×3=1200 EPS，平台 wl_limit 接受率仅 ~1.7%，EPS 曲线高低不均，连心跳专用主机也出现“服务端关闭”断线。

**Bug 1：HeartbeatWorker 无锁写入 → TCP 流数据混乱（根因）**

v3.7.27 的 HB 写入代码：
```csharp
bool lockAcquired = false;
if (_streamRegistry != null)
    lockAcquired = await AcquireWriteLockAsync(...);

await stream.WriteAsync(payload);
```
当写锁获取超时（LogWorker 正在写同一 stream），HB 仍然直接写入，导致两个 Writer **并发写同一 TCP 流**，PT 包字节交错 → 平台收到垃圾数据 → 关闭连接。蓝屏投影效应波及其他主机的纯心跳客户端。

`stream.WriteAsync(payload, ct)` 的 ct 取消时 .NET 会 RST socket（发 TCP RST 而非 FIN），导致平台异常关闭连接。这是 v3.7.3 Fix 1 的回归。

**修复**：`WriteAsync` 不传 CT，背压时由 TCP KeepAlive 检测死连接。

### v3.7.25 — 2026-03-18：根本修复（第二处）——SessionStale 误判断线导致 2min 振荡

**根因（分析 PolicyReceiveWorker + 老工具 C++ 行为对照）**：

`onDataReceived` 在每次 `ReadAsync` 收到**任意数据**时即触发，`lastReplyTimeMs` 值准确无误。据此可判定：平台从未向 79% 的客户端发过任何数据（`lastReply == 0`）。

SessionStale 的旧逻辑：
```
refTime = lastReply > 0 ? lastReply : connectedAtMs  ← 错误！把"平台忙"误判为"session 失效"
```
对 `lastReply == 0` 的客户端，`refTime = connectedAtMs`，90s 后必然断线。

结果：
- 79% 客户端每 90s SessionStale 一次→ Unregister → 30s 重连等待 → 重连后仍无回包 → 90s 后再断
- 振荡周期 = 90 + 30 = **120s ≈ 2 分钟**，与 EPS 图吻合
- 老工具 `RecvHeartBeatBack_TCP` 2s 超时后直接继续，**永不因"未收到回包"主动断线**

**修复**：条件改为 `lastReply > 0 && (now - lastReply) > staleMs`。`lastReply == 0`（本次连接从未收到回包）时不处理 SessionStale，平台需要时自己会发 FIN。

### v3.7.24 — 2026-03-18：根本修复（第一处）——断线期间静默丢弃日志包，消除重连 burst

**根因（最终定位，C++ 源码对照）**：

老工具 Log 线程持有初始连接时的 `pHeapArgs->sock`（来自 `g_sock[i]`）。HB 线程调用 `CreateConnection` 重连后，`g_sock[i]` 全局数组从未更新，Log 线程自始至终用的是 stale socket。断线期间 `send()` 在 stale socket 上直接返回 `SOCKET_ERROR`，每包静默失败，Channel 中无任何积压。重连后从零开始发送，不产生 burst，平台不触发 FIN。

新工具 Channel 架构下，断线期间 `TryEnqueueLog` 不判断连接状态，仍然成功入队。30 s 相位等待窗口内可积压 ~90 包（3 包/s × 30 s），重连后 drain 循环瞬间清空，造成 burst → 平台发 FIN → 新一轮断线 → 循环震荡。这同时解释了：

- **持续断线**：burst 触发平台 FIN  
- **EPS 震荡**：burst 后队列清空，短暂 EPS 归零再回升  
- **EPS 低于理论值**：burst 期间部分包被平台丢弃或限速

**修复**：在 `HeartbeatStreamRegistry.TryEnqueueLog` 中，入队前先检查 `_streams.ContainsKey(clientId)`；若 stream 未注册（即客户端处于断线状态），直接返回 `"disconnected"`，静默丢弃该包。LogWorker 已对非 null 返回值走失败统计并 break，无需改动。

### v3.7.21 — 2026-03-18：相位对齐重连，彻底对齐老工具 C++ 重连散布机制

**根因（经 C++ 源码对照分析）**：

旧工具 `ThreadFunc_HeartbeatSend_New` 的循环结构：
```
CreateConnection → SendHB → RecvHBBack → Sleep(30000) → [循环]
```
`Sleep(30000)` **始终执行**，无论本轮是否出现断线。平台批量发 FIN 时，各客户端正处于不同的 Sleep 剩余时间（因启动时 500ms×50批 ≈ 25s 错开），重连时刻天然散布在整个 30s 窗口，不需要任何 jitter 代码。

新工具两个问题未对齐：
1. **spreadMs 上限写死 3000ms**：500 个客户端仅在 3s 内错开（每隔 6ms），几乎同相；老工具覆盖整个 30s 窗口。
2. **WriteFailed/ServerClosed 后立即 `continue` 重连**：跳过了等价于 Sleep(30000) 的 drain 等待期。所有客户端收到 FIN 后在毫秒级内全部冲击平台，形成雪崩。

v3.7.20 加的 0-2s / 0-5s 随机 jitter 方向没错，但窗口远小于 30s，且每次随机会破坏客户端的相位（导致下次断线时再次堆叠）。

**修复方案**：
- `spreadMs = intervalMs`：500 个客户端相位均匀散布在完整 30s 窗口
- `nextCycleTargetMs`：每次 HB 写入后记录 `now + intervalMs`；断线后重连等待直到此时刻，完整保留相位差
- 移除所有随机 jitter（ServerClosed/WriteFailed 0-2s、SessionStale 0-5s），由相位门控替代

**变更文件**：
| 文件 | 变更 |
|---|---|
| `HeartbeatWorker.cs` | `spreadMs=intervalMs`；新增 `nextCycleTargetMs` 变量；重连块改为相位等待；HB 写入成功后设置 `nextCycleTargetMs`；WriteFailed 时设置 `nextCycleTargetMs` 再 `continue`；删除 SessionStale 随机 jitter |

### v3.7.20 — 2026-03-18：修复 ServerClosed/WriteFailed 批量断线雪崩重连 + drain CTS 复用 bug

**现象**（v3.7.19）：EPS 仍有尖峰，大量"服务端关闭"断线；无日志发送失败；平台 CPU/内存正常（56%/69%）。

**根因**：EPS 本身不是问题（平台能轻松处理 6000 EPS）。v3.7.19 中"目标 300 EPS"是错误假设，限速方向错了。真正的问题是：

1. **雪崩重连（thundering herd）**：平台在 session 清理时会批量发送 FIN（一次 60 个），所有收到 FIN 的客户端立即同时发起 ConnectAsync，形成连接请求冲击，加剧断线循环。SessionStale 已有 0-5s jitter，但 ServerClosed/WriteFailed 没有任何延迟。

2. **drain CTS 复用 bug**：`using var drainWriteCts` + `TryReset()` 模式：当某次写超时（3s），CTS 被取消，`TryReset()` 返回 false，此后所有写操作立即以 OperationCanceledException 失败，导致连接被销毁（虽然 TCP 本身没问题）。

**修复**：

- **ServerClosed/WriteFailed 重连加 0-2s 随机 jitter**：60 个客户端分散在 0-2s 内重连，而非同时冲击平台
- **drain CTS 改为每次独立创建**：`using var writeCts = CancellationTokenSource.CreateLinkedTokenSource(ct); writeCts.CancelAfter(3000)`，每次 `using` 保证正确释放，彻底消除状态复用问题

| 文件 | 变更 |
|---|---|
| `HeartbeatWorker.cs` | 在 `hasConnectedBefore && ServerClosed/WriteFailed` 时加 `Random.Shared.Next(0,2000)ms` jitter；drain 写 CTS 改为循环体内 `using var` 独立创建 |

### v3.7.16 — 2026-03-18：Channel 架构——HeartbeatWorker 统一写 TCP，彻底消除锁竞争

**根因**（所有 v3.7.x 修复均未触及）：

> 应用层写锁（SemaphoreSlim）本身是错误的抽象。老工具 C++ 的 HeartbeatThread + MsgLogThread 并发 `send(g_sock[i])`，依靠 OS Winsock 在内核层串行化——应用层零锁。C# 对等实现是「单一写入方」，而非「加锁多写方」。

| 版本 | 根因 | 表现 |
|---|---|---|
| v3.7.11 | 4000ms 超时 = 集体节拍器 | 400 客户端同步 skip → 振荡 |
| v3.7.13 | `lock` 阻塞线程池线程 | 400 线程卡住 → 线程池饥饿 |
| v3.7.14 | 无超时等锁，WriteAsync 背压永久阻塞 | 心跳饿死 → 平台 FIN |
| v3.7.15 | 5s 超时→强制重连 | 治标，重连开销仍存在 |
| **v3.7.16** | **Channel 架构，无锁** | **HeartbeatWorker 唯一写方，无竞争** |

**新架构**：

```
[LogWorker]  →  TryEnqueueLog(clientId, payload)  →  Channel<byte[]>  →  [HeartbeatWorker]
                  (非阻塞，DropOldest)                   per clientId       drain in HB gap
```

- **HeartbeatWorker**：写完 HB 后，在 `intervalMs - 100ms` 窗口内 drain 日志 Channel 并写 TCP。
  - HB 写入完全不受日志影响，心跳时序铁板一块。
  - TCP 背压只影响日志 drain 速率，不影响 HB。
- **LogWorker**：`TryEnqueueLog()` → `Channel.Writer.TryWrite()`，非阻塞，立即返回；队列满时自动丢弃最旧日志（压力测试场景可接受）。
- **HeartbeatStreamRegistry**：移除 `SemaphoreSlim` 写锁，改为 `Channel<byte[]>`（容量 2000，DropOldest）；新增 `TryEnqueueLog` / `GetLogReader` API。

| 文件 | 变更 |
|---|---|
| `HeartbeatStreamRegistry.cs` | 完全重写：SemaphoreSlim → Channel，移除所有锁 API |
| `HeartbeatWorker.cs` | 移除锁获取逻辑；HB后进入 drain 循环（替代 wakeupSem.WaitAsync） |
| `LogWorker.cs` | `SendLogTcpLikeExternalAsync`（async TCP写）→ `TryEnqueueLog`（同步入队） |

### v3.7.15 — 2026-03-18：锁等待超时→强制重连（背压下保证心跳）

### v3.6.0 — 2026-03-07：TCP 发送可靠性修复（对齐老工具行为）

**背景**：实测发现 C# 工具在 1000 客户端 + 1 EPS 场景下日志发送能力弱于老工具（C++），深度代码对比发现 6 处行为偏差。

**修复内容：**

1. **\[P0\] `write_ex` 触发 Unregister**（`LogWorker.SendLogTcpLikeExternalAsync`）
   - 旧：`WriteAsync` thrown 后返回 `write_ex:*`，stream 保留在注册表；心跳下一次写入被污染的半截 stream → 平台断连 → 最多要等 90s SessionStale 才能重连。
   - 新：catch 块中立即 `_streamRegistry.Unregister(clientId)`，HeartbeatWorker 的 drain 任务收到 `n==0` 后在 1~2 个心跳周期内完成重连。

2. **\[P0\] `allThreatTcp` 失败后立即 break**（`LogWorker.StartCoreAsync`）
   - 旧：5 种类型逐一发送，任意一种 `stream_null` 后仍继续发剩余类型并等待 50ms×N，导致失败计数 ×5 倍膨胀，并浪费 200ms。
   - 新：对齐 C++ `goto END`，任一类型失败后立即 `break`，剩余类型不再发送。

3. **\[P1\] LockBusy 时补发心跳而非跳过**（`HeartbeatWorker`）
   - 旧：4000ms 等锁超时后直接 `continue`，整个 30s 周期内少发 1 次心跳（实际变为 60s 无心跳）。
   - 新：等 200ms 后再重试获取锁（500ms 超时），锁成功则立即补发；仍获取失败才跳过本次，且不等完整 intervalMs。

4. **\[P1\] SessionStale 重连加随机 jitter**（`HeartbeatWorker`）
   - 旧：270 个客户端同时触发 SessionStale 后同时发起 `ConnectAsync`，冲击平台 → 更多拒绝 → 雪崩。
   - 新：重连前加 `Random.Shared.Next(0, 5000)` ms 随机延迟，打散连接潮。

5. **\[P3\] drain 异常记录具体类型**（`HeartbeatWorker`）
   - 旧：所有 drain 异常统一记为 `reason=连接异常`，无法区分网络错误/正常 Dispose/取消。
   - 新：`catch (Exception exDrain)` 记录 `exDrain.GetType().Name`；`ObjectDisposedException` 单独映射为 `连接已释放`。

6. **\[P3\] `evtDisconnAtMs[]` 改用 `Interlocked.Exchange`**（`HeartbeatWorker`）
   - 旧：`long` 类型在 32-bit 运行时不保证原子写；drain Task 和主循环双写同一元素存在数据竞争。
   - 新：所有写入点（PolicyDrain / RawDrain / drain catch / ConnTimeout / ConnFailed / WriteFailed / SessionStale）均改为 `Interlocked.Exchange`。

### v3.6.1 — 2026-03-08：修复 stream_null 永久空缺（压测日志分析发现）

**根因**：1 天压测日志分析发现，TCP 风暴期间 LogWorker 写入超时 → `write_ex:OperationCanceledException` → 调用 `Unregister`；但此时 HeartbeatWorker 的 TCP 仍然存活，`if (tcp == null || !alive)` 为 false，永远不走重连分支，因此也永远不再执行 `Register`。注册表空缺持续至该 TCP 自然死亡（约 5 小时），期间所有日志静默丢弃。

**现象**：平台看到心跳全部在线，但 200 客户端中约 60 个日志静默，EPS 从期望 1000 降至 ~700，持续 5 小时（`stream_null` 586,266 次，~34.5 次/秒，`lock_timeout` 2,181 次）。

**修复**：在 HeartbeatWorker `WriteAsync + FlushAsync` 成功后、`ReleaseWriteLock` 之前调用 `_streamRegistry?.Register(c.ClientId, stream!)`。
- 位于**写锁内**：确保 LogWorker 等锁方在锁释放后能立即 `TryGet` 成功，消除 ReleaseWriteLock 与 Register 之间的竞态窗口
- **幂等**：`ConcurrentDictionary[key] = value`，不影响正常连接流程  
- 最迟下一心跳周期（≤30s）恢复，而非等待数小时 TCP 自然死亡

| 文件 | 变更 |
|---|---|
| `src/SimulatorLib/Workers/HeartbeatWorker.cs` | 心跳写成功后幂等重注册 stream（锁内，消除竞态） |

### v3.3.0 — 2026-03-04：修复锁超时导致心跳 TCP 被错误销毁

**问题**：`HeartbeatWorker` 在写锁等待超时（`lockAcq=false`）时，原代码抛出 `TimeoutException` 被外层 `catch` 误判为 TCP 写入失败，执行 `tcp.Dispose()` 销毁了仍然有效的 TCP 连接。后续 HeartbeatWorker 重连因平台连接槽被 LogWorker 降级独立 TCP 占满而失败，最终所有客户端心跳离线，但日志还在持续发送。

**触发场景**：多台机器同时高速打日志（如 3台 × 100客户端 × 13条/s），平台侧繁忙导致 `WriteAsync` 阻塞，`sendCts`（5s总预算）耗尽后 `AcquireWriteLockAsync` 返回 `false`。

**雪崩链路**：`锁超时 → tcp.Dispose() → HeartbeatWorker重连 → LogWorker降级独立TCP → 平台连接槽满 → 重连永久失败 → 心跳全离线但日志仍在发`

**修复**：`lockAcq=false` 时跳过本次心跳（`continue`），不 Dispose TCP。新增 `Reason.LockBusy = 6`，不计入离线数，在监控日志和 UI 中显示为 `⚡锁竞争跳过:N`。

| 文件 | 变更 |
|---|---|
| `src/SimulatorLib/Workers/HeartbeatWorker.cs` | `lockAcq=false` 时跳过心跳而非 Dispose TCP；新增 `Reason.LockBusy`；统计/上报加入 LockBusy 计数 |
| `src/SimulatorApp/ViewModels/MainViewModel.cs` | UI 日志显示 `⚡锁竞争跳过:N`；`HeartbeatStats` 新增 `RsnLockBusy` 字段 |

### v3.2.0 — 2026-03-03：NetworkStream 写锁

**问题**：v3.1.0 复用心跳 TCP stream 后，`HeartbeatWorker`（定时写心跳）与 `LogWorker`（按 EPS 写日志）**并发 `WriteAsync` 同一 `NetworkStream`**，导致：
- PT 包体字节互相穿插 → 平台主动断连 → **3000 客户端全部离线**
- 策略回包损坏 → 平台策略审计始终显示 **"待发送"**

**修复**：`HeartbeatStreamRegistry` 为每个 clientId 维护 `SemaphoreSlim(1,1)` 写锁，心跳写与日志写串行化。

| 文件 | 变更 |
