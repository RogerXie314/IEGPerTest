```powershell

dotnet build

# 2. 本地调试：启动本地接收器（可选，用于离线验证心跳是否到达）

dotnet run --project src/SimulatorApp

.\artifacts\SimulatorAppPublish\SimulatorApp.exe
```

```powershell

```

## 🔬 问题调查记录（待验证）

### 现象（2026-03-03 测试）

截图与日志（`C:\Users\Roger\Downloads\log\`）显示：
- 1000 客户端已连接，心跳任务已发送 1000 次，平台回包(90s窗口)**仅 57**，随后持续为 0
- 日志：`TCP连接中但未回包:1000 ← 平台静默踢出则数量偏高`
- 启动日志任务（操作系统日志 IEG，3 EPS × 1000 客户端）约 **2 分钟后** 平台开始踢 session

### 推测原因 A：HeartbeatWorker 写锁超时后仍然写入（代码 bug）

`HeartbeatWorker.cs` 第 235–244 行：

```csharp
bool lockAcq = _streamRegistry != null
    ? await _streamRegistry.AcquireWriteLockAsync(c.ClientId, 4000, sendCts.Token)
    : true;
try
{
    await stream!.WriteAsync(payload, ...);  // ← lockAcq=false 时未跳过，锁形同虚设
```

`AcquireWriteLockAsync` 超时返回 `false` 后，代码没有 `if (!lockAcq) throw/skip` 的保护，心跳依然写入，写锁失效。

**验证方式**：在高 EPS 场景下加日志打印 `lockAcq=false` 的频率，确认是否频繁超时。

**修复方向**：超时未获锁时应抛出异常（触发重连流程）或至少跳过本次写入。

### 推测原因 B：平台对单条 TCP 连接上的高速 CMDID=21 包有限速/拒绝策略

3 EPS × 1000 客户端 = **3000 包/秒** 并发写入各自的心跳 stream。即使写锁正确，平台处理层可能对心跳连接上同时出现的大量 CMDID=21 日志包启动了速率限制或 session 踢除策略（原始 C++ 测试时 EPS 规模未知）。

**验证方式**：
- **取消连接复用**，改用独立 TCP 短连接发 CMDID=21 日志，观察心跳是否恢复正常 ACK
- 如心跳恢复正常 → 说明平台对"心跳连接上发日志"有限制，需考虑方案调整
- 如心跳仍异常 → 问题在客户端侧写锁或并发本身

### 下一步建议（在公司环境中验证）

1. 先修复写锁超时未保护的 bug（推测原因 A），重新测试
2. 若仍离线，临时关闭 stream 复用（`_streamRegistry = null`），改用独立连接发威胁日志，测试心跳在线率
3. 对照原始 IEG C++ 代码，确认 `SetWLTcpConnectInstance` 注入后是否有额外的"心跳连接只发心跳"的互斥逻辑

## 📝 变更记录

### v3.2.0 — 2026-03-03：NetworkStream 写锁

**问题**：v3.1.0 复用心跳 TCP stream 后，`HeartbeatWorker`（定时写心跳）与 `LogWorker`（按 EPS 写日志）**并发 `WriteAsync` 同一 `NetworkStream`**，导致：
- PT 包体字节互相穿插 → 平台主动断连 → **3000 客户端全部离线**
- 策略回包损坏 → 平台策略审计始终显示 **"待发送"**

**修复**：`HeartbeatStreamRegistry` 为每个 clientId 维护 `SemaphoreSlim(1,1)` 写锁，心跳写与日志写串行化。

| 文件 | 变更 |
|---|---|
| `src/SimulatorLib/Workers/HeartbeatStreamRegistry.cs` | 新增 `_writeLocks`、`AcquireWriteLockAsync`、`ReleaseWriteLock` |
| `src/SimulatorLib/Workers/HeartbeatWorker.cs` | 心跳 `WriteAsync` 前获取写锁，finally 释放 |
| `src/SimulatorLib/Workers/LogWorker.cs` | 共享 stream 写入前获取写锁，重取最新 stream，finally 释放 |

---

### v3.1.0 — 2026-03-03：日志发送复用心跳 TCP 流

**问题**：日志发送为每客户端另开独立 TCP 连接，导致同一 ComputerID 存在两条连接，平台踢掉后发者。

**修复**：引入 `HeartbeatStreamRegistry`，`LogWorker` 优先复用心跳 stream，降级才走独立连接。

| 文件 | 变更 |
|---|---|
| `src/SimulatorLib/Workers/HeartbeatStreamRegistry.cs` | 新增（stream 注册表） |
| `src/SimulatorLib/Workers/LogWorker.cs` | 复用心跳 TCP 流、任务面板集成 |
| `src/SimulatorLib/Workers/HeartbeatWorker.cs` | 回包统计修复（每次 ReadAsync 立即更新时间戳）、TCP 重连对齐原始 2s 超时 |

### v3.0.0 — 2026-03-03：任务面板 + 策略接收 + 威胁检测子类

1. **威胁检测拆分为 5 种事件子类**（TCP 长连接，CMDID=21，支持 6000 EPS）：`ThreatRegAccess`、`ThreatFileAccess`、`ThreatOsEvent`、`ThreatDllLoad`

3. **策略接收**：`PolicyReceiveWorker` 在心跳 TCP 流上监听下发包，解析 CMDTYPE/CMDID，HTTPS 回 ACK
4. **白名单轮换重写**：`RunRotatingAsync`，随机轮换，15 分钟周期，TaskRecord 集成
5. **注册版本选择**：ComboBox 选择客户端版本（默认 `V300R011C01B030`）
6. **分通道独立配置**：HTTPS 与威胁检测各自独立配置客户端数和 EPS
7. **失败日志按文件大小（50 MB）限制**，移除硬编码条数上限
