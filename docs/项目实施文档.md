## 🔬 问题调查记录

### ✅ 已解决：心跳离线但日志仍在发（2026-03-04）

**现象**：100台×13条/s 运行一段时间后，心跳全部离线，但平台后台日志(cmdId:200)还在持续流入。包括完全没有发日志、只开心跳的另外两台机器也全部离线。

**根本原因**：心跳写锁超时后错误地 `Dispose` 了仍然有效的 TCP 连接，触发重连雪崩。

雪崩链路：
```
LogWorker 高速写 → WriteAsync 因平台繁忙阻塞 → HeartbeatWorker 等锁超时
  → lockAcq=false → throw TimeoutException（旧代码）
  → 外层 catch 误判为写失败 → tcp.Dispose()（TCP 其实没坏！）
  → HeartbeatWorker 重连 → LogWorker 降级建独立 TCP
  → 平台连接槽(ulimit=1024)被独立 TCP 占满
  → 所有机器 HeartbeatWorker 重连失败 → 全部永久离线
  → 但 LogWorker 独立 TCP 继续发日志 → 平台看到日志来但心跳消失
```

**修复**（v3.3.0）：`lockAcq=false` 时跳过本次心跳并 `continue`，不 Dispose TCP。详见变更记录。

## 📝 变更记录

### v3.3.0 — 2026-03-04：修复锁超时导致心跳 TCP 被错误销毁

**问题**：`HeartbeatWorker` 在写锁等待超时（`lockAcq=false`）时，原代码抛出 `TimeoutException` 被外层 `catch` 误判为 TCP 写入失败，执行 `tcp.Dispose()` 销毁了仍然有效的 TCP 连接。后续 HeartbeatWorker 重连因平台连接槽被 LogWorker 降级独立 TCP 占满而失败，最终所有客户端心跳离线，但日志还在持续发送。

**触发场景**：多台机器同时高速打日志（如 3台 × 100客户端 × 13条/s），平台侧繁忙导致 `WriteAsync` 阻塞，`sendCts`（5s总预算）耗尽后 `AcquireWriteLockAsync` 返回 `false`。

**雪崩链路**：`锁超时 → tcp.Dispose() → HeartbeatWorker重连 → LogWorker降级独立TCP → 平台连接槽满 → 重连永久失败 → 心跳全离线但日志仍在发`

**修复**：`lockAcq=false` 时跳过本次心跳（`continue`），不 Dispose TCP。新增 `Reason.LockBusy = 6`，不计入离线数，在监控日志和 UI 中显示为 `⚡锁竞争跳过:N`。

| 文件 | 变更 |
|---|---|
| `src/SimulatorLib/Workers/HeartbeatWorker.cs` | `lockAcq=false` 时跳过心跳而非 Dispose TCP；新增 `Reason.LockBusy`；统计/上报加入 LockBusy 计数 |
| `src/SimulatorApp/ViewModels/MainViewModel.cs` | UI 日志显示 `⚡锁竞争跳过:N`；`HeartbeatStats` 新增 `RsnLockBusy` 字段 |

### v3.2.0 — 2026-03-03：NetworkStream 写锁

**问题**：v3.1.0 复用心跳 TCP stream 后，`HeartbeatWorker`（定时写心跳）与 `LogWorker`（按 EPS 写日志）**并发 `WriteAsync` 同一 `NetworkStream`**，导致：
- PT 包体字节互相穿插 → 平台主动断连 → **3000 客户端全部离线**
- 策略回包损坏 → 平台策略审计始终显示 **"待发送"**

**修复**：`HeartbeatStreamRegistry` 为每个 clientId 维护 `SemaphoreSlim(1,1)` 写锁，心跳写与日志写串行化。

| 文件 | 变更 |
|---|---|
| `src/SimulatorLib/Workers/HeartbeatStreamRegistry.cs` | 新增 `_writeLocks`、`AcquireWriteLockAsync`、`ReleaseWriteLock` |
| `src/SimulatorLib/Workers/HeartbeatWorker.cs` | 心跳 `WriteAsync` 前获取写锁，finally 释放 |
| `src/SimulatorLib/Workers/LogWorker.cs` | 共享 stream 写入前获取写锁，重取最新 stream，finally 释放 |

### v3.1.0 — 2026-03-03：日志发送复用心跳 TCP 流

**问题**：日志发送为每客户端另开独立 TCP 连接，导致同一 ComputerID 存在两条连接，平台踢掉后发者。

**修复**：引入 `HeartbeatStreamRegistry`，`LogWorker` 优先复用心跳 stream，降级才走独立连接。

| 文件 | 变更 |
|---|---|
| `src/SimulatorLib/Workers/HeartbeatStreamRegistry.cs` | 新增（stream 注册表） |
| `src/SimulatorLib/Workers/LogWorker.cs` | 复用心跳 TCP 流、任务面板集成 |
| `src/SimulatorLib/Workers/HeartbeatWorker.cs` | 回包统计修复（每次 ReadAsync 立即更新时间戳）、TCP 重连对齐原始 2s 超时 |
