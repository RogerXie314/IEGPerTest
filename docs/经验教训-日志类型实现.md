# 经验教训：日志类型实现问题排查

## 问题概述
**问题**：文件保护、注册表保护、强制访问控制三个日志类型无法在平台显示  
**根本原因**：LogContent 字段格式错误（使用了描述性中文文本，而平台要求简单数字字符串）  
**解决时间**：2026-02-11  
**循环轮数**：约11轮排查

---

## 排查过程回顾

### 第1-2轮：字段缺失假设 ❌
- **假设**：HostDefence日志缺少 ClientIP/ClientName/MachineCode/OS 字段
- **行动**：添加了4个额外字段
- **结果**：问题未解决

### 第3-4轮：字段数量纠正 ❌
- **发现**：WLDefender.log 实际只有7个字段
- **行动**：移除额外字段，恢复7字段结构
- **结果**：JSON结构正确，但问题依然存在

### 第5-6轮：代码路由研究 ❌
- **怀疑**：IsHttpsOnlyCategory 路由逻辑不一致
- **行动**：深入研究 AccessServer.cpp 的路由代码
- **结果**：路由逻辑正确，未找到问题

### 第7-8轮：C++源码分析 ❌
- **方向**：分析 hostDefence_GetJsonByVector 实现
- **行动**：阅读 WLJsonParse.h 和相关头文件
- **结果**：未找到JSON生成的特殊逻辑

### 第9-10轮：前置日志误导 ❌
- **观察**：WLDefender.log 中 HostDefence 日志前有 Admin 管理日志
- **假设**：需要先发送管理日志才能发送 HostDefence 日志
- **澄清**：用户确认管理日志是模块启用产生的，不是必须的前置条件

### 第11轮：精确字段格式对比 ✅
- **方法**：逐字段对比 WLDefender.log 中的实际值
- **发现**：LogContent 字段格式错误
  ```json
  // 原项目实际格式
  {"LogContent":"1", ...}
  {"LogContent":"2", ...}
  {"LogContent":"3", ...}
  
  // 我们的错误格式
  {"LogContent":"文件[C:\\Temp\\...]被notepad.exe非法修改", ...}
  ```
- **修正**：改为简单数字字符串
  ```csharp
  logContent: ((messageIndex % 5) + 1).ToString()
  ```
- **结果**：✅ 问题完全解决，所有3个日志类型正常显示

## 核心教训

### 1. **数据优先于代码** ⭐⭐⭐
**错误做法**：先研究代码逻辑、路由机制、C++实现  
**正确做法**：先完整对比实际日志数据的每个字段的值格式

**对比检查清单**：
- [ ] 字段名称是否完全一致
- [ ] 字段数量是否匹配
- [ ] 字段顺序是否一致
- [ ] **字段值格式是否精确匹配** ← 本次遗漏的关键点
- [ ] 数据类型（字符串/数字/布尔）是否正确

### 2. **平台侧验证严格性**
即使 JSON 结构完全正确，字段值格式不符合预期也会导致日志被拒绝。平台侧可能有：
- 字段值格式验证（如 LogContent 只接受数字字符串）
- 字段值范围限制
- 字段值正则表达式匹配

### 3. **避免过早假设**
**过早假设的代价**：
- 浪费时间研究不相关的代码路径
- 忽略简单直接的数据对比
- 陷入复杂的理论推导而非实证检查

**正确的心态**：
- 先用数据说话，再看代码
- 简单解释优先（Occam's Razor）
- 每个字段都要验证，不放过任何细节

### 4. **正确的排查顺序**
```
优先级1: 完整对比实际日志的每个字段的值格式
         ↓
优先级2: 验证 JSON 结构和字段顺序
         ↓
优先级3: 检查 endpoint 路径和 CMDID
         ↓
优先级4: 验证 HTTPS/UDP 路由逻辑
         ↓
优先级5: 研究 C++ 源码实现细节
```

### 5. **特定于 HostDefence 日志的发现**
- LogContent 字段：必须是简单数字字符串（"1", "2", "3" 等）
- 不能使用描述性文本
- 平台侧有严格的格式验证
- 7个字段：Block、FullPath、LogContent、LogType、ProcessName、Time、Username

## 后续建议

### 实现新日志类型时的检查清单
1. **找到实际日志样例**（WLDefender.log 或平台抓包数据）
2. **逐字段对比**：
   - 字段名（大小写）
   - 字段值格式（数字/字符串/布尔/对象）
   - 字段值内容示例
   - 字段顺序
3. **验证 HTTPS endpoint**：
   - 路径完全匹配 commurl.h
   - CMDID 正确
4. **测试验证**：
   - 先发送单条日志测试
   - 检查平台响应状态码
   - 启用 TryDebugWriteHttps 记录详细日志

### 避免的陷阱
- ❌ 看到 JSON 结构正确就认为没问题
- ❌ 只关注字段名称，忽略字段值格式
- ❌ 过早深入研究 C++ 源码
- ❌ 依赖代码推理而非实际数据验证

### 推荐工具
- JSON 差异对比工具（如 jsondiffpatch）
- Wireshark/Fiddler 抓包验证实际发送的数据
- 启用应用内的调试日志（TryDebugWriteHttps）
